# Задание на доработку лендинга: прогнозная карта, новые визуализации, проверяемая исследовательская система

Статус: рабочее задание для исследовательской группы (ИГ) и команды разработки (РК).
Дата: 2026-07-11. Версия: 1.0.
Основание: [ROADMAP_FORECAST.md](ROADMAP_FORECAST.md) (принят).
Связанные документы: [ARTIFACT_STANDARD.md](ARTIFACT_STANDARD.md) (стандарт
проверяемого пакета), [METHODOLOGY.md](METHODOLOGY.md), [SOURCES.md](SOURCES.md).

## 0. Цель, рамки, принципы

**Цель.** Превратить лендинг из набора визуализаций в прозрачную
исследовательскую систему: каждый значимый вывод можно проследить до
источника, воспроизвести и независимо перепроверить — вручную или
LLM-агентом пользователя.

**Состав работ.** Р1 — прогнозная карта 2026–2075; Р2 — девять новых
инфографик; Р3 — методологическая прозрачность каждой публикации;
Р4 — стандарт и конвейер проверяемых ZIP-пакетов; Р5 — раздел лендинга
«Проверяемые артефакты»; §6 — приоритетный план; §7 — критерии приёмки.

**Вне рамок этапа 1**: вероятностный слой прогноза на уровне районов;
серверный бэкенд (сохраняем статический экспорт Next.js); автоматическое
обновление данных чаще раза в год.

**Сквозные принципы.**

1. Прогноз и аналитика не создают иллюзию точности: чем дальше горизонт и
   меньше территория — тем шире интервалы, вплоть до перехода от чисел к
   классам динамики.
2. Ни одна инфографика не публикуется без методологического блока (Р3) и
   проверяемого пакета (Р4). Это гейт в CI, а не пожелание.
3. Все сырые данные вендорятся в `data/raw/` с датой обращения и лицензией —
   немедленно при получении (источники могут закрыться).
4. Каждое значение в данных несёт тип достоверности; прогноз добавляет тип
   `f` с обязательными атрибутами «сценарий» и «версия прогноза».
5. Нейтральный тон, проверяемые источники, явные оговорки — стиль проекта.

**Роли в карточках работ:** ИГ — данные, модели, методика, тексты;
РК — ETL-интеграция, UI, CI, пакетный конвейер; РЕД — редакторская и
методологическая вычитка (второй человек, не автор).

### WP-F2. Модельный конвейер `etl/forecast/` (ИГ+РК)

Модели по уровням (детали и ссылки — roadmap §1–2, §6):

- страна и области + Минск: когортно-компонентный метод (CCMPP), траектории
  СКР/ОПЖ — байесовские (`bayesTFR`/`bayesLife`, медиана + квантили 10/90);
  кросс-проверка смертности моделью Ли–Картера на HMD;
- 118 районов: Гамильтон–Перри (CCR 2009→2019) со shrinkage малых районов к
  областному профилю и трендированием CCR к областному CCMPP;
- 222 города: логистическая модель доли города в районе; Минск и 5
  облцентров — собственные CCMPP;
- внутренняя миграция: гравитационная модель, калибровка на матрице
  2010–2019; международная — сценарные допущения (WP-F4);
- согласование: IPF по (год × пол × возрастная группа): районы→область,
  области+Минск→страна, город ≤ район.

Предлагаемая структура модуля:

```
etl/forecast/
  jumpoff.py      # стартовые структуры 2026 (два ряда: official/adjusted)
  fertility.py    # траектории СКР (обёртка bayesTFR или калиброванные кривые)
  mortality.py    # e0 + Ли-Картер
  migration.py    # гравитационная модель + международные сценарии
  ccmpp.py        # когортная продвижка (страна/области/крупные города)
  ccr.py          # Гамильтон-Перри для районов
  reconcile.py    # IPF-согласование уровней
  scenarios.py    # параметры сценариев (yaml) -> прогоны
  backtest.py     # ретро-валидация (WP-F5)
  export.py       # forecast.json
  tests/
```

R-зависимости (bayesTFR/bayesLife) допустимы только на этапе подготовки
траекторий с фиксацией результатов в `data/curated/trajectories/`;
воспроизводимый прогон самого прогноза — чистый Python (как весь ETL).

Приёмка: юнит-тесты на каждую модель; инварианты (суммы сходятся после IPF,
нет отрицательных населений, население когорты не растёт без миграции);
полный прогон `python -m etl.forecast` детерминирован (фиксированный seed).

### WP-F3. Корректировка официальной статистики (ИГ)

По roadmap §5: реконструировать интервальную оценку неучтённого оттока
2020–2026 из зеркальной статистики (дедупликация, досчёт ненаблюдаемых
направлений по опросным долям), построить **два стартовых ряда** 2026 г. —
`official` (ряд Белстата) и `adjusted` (минус интервальная поправка с
временным профилем и территориальным распределением: стартовые ключи —
Минск 55–70%, облцентры 20–30%, остальное — крупные райцентры).

Результаты: `data/curated/adjustment.csv` (год × территория × поправка
low/mid/high) + аналитическая записка `docs/notes/adjustment.md` с
чувствительностью итогов прогноза к поправке.

Приёмка: каждая компонента поправки привязана к источнику с датой; интервал
не уже официально опубликованных ориентиров (совокупный отток ≥ стока ВНЖ ЕС
за вычетом довоенного стока); записка вычитана РЕД.

### WP-F4. Сценарии (ИГ)

Три детерминированных сценария (`base`, `optimistic`, `negative`) —
стартовые параметры в таблице roadmap §7 (СКР, ОПЖ, международное сальдо,
центростремительность, стартовый ряд). Параметры хранятся в
`etl/forecast/scenarios/*.yaml` с обоснованием каждого значения (комментарий
+ ссылка).

Калибровка: национальный итог `base` на 2050 г. — в пределах ±3% от медианы
UN WPP 2024 (7,45 млн); `optimistic` — в коридоре high-варианта WPP;
`negative` — ниже low-варианта WPP с обоснованием (миграционный шок, который
ООН для РБ не закладывает). Расхождения фиксируются в записке, а не
подгоняются молча.

### WP-F5. Валидация (ИГ, гейт для публикации)

1. **Бэктест**: обучение на данных ≤2009 → прогноз 2019 → сравнение с
   переписью-2019 по 118 районам и 222 городам. Метрики: MAPE, доля
   территорий с верным знаком динамики, для интервалов — покрытие.
   Бенчмарк — наивная линейная экстраполяция 1999–2009.
   **Гейт: модель обязана превосходить бенчмарк по MAPE на уровне районов и
   городов; национальный итог бэктеста — в пределах ±2% факта.**
2. **Сверка агрегатов**: таблица «наш base vs WPP medium vs WCDE SSP2» по
   2030/2040/2050/2075; все расхождения >5% объяснены письменно.
3. **Чувствительность**: вариация ±0,1 СКР, ±1 год ОПЖ, ±50% миграционной
   поправки → таблица эластичностей итогов 2050/2075.
4. Отчёт `docs/notes/validation.md` — обязательная часть пакета артефактов
   прогноза.

### WP-F6. Продукт и UI (РК)

**Формат данных** — отдельный `web/public/data/forecast.json` (историческую
`data.json` не трогаем):

```jsonc
{
  "version": "v2026.1",            // версия прогноза, попадает в тултипы и цитирование
  "horizon": [2026, 2075],
  "scenarios": ["base", "optimistic", "negative"],
  "jumpoff": ["official", "adjusted"],   // этап 3
  "territories": {
    "BY": { "base": { "years": [2026, 2030, ...],
                       "pop":   [9109000, ...],
                       "q10":  [...], "q90": [...] }, ... },
    "r-drybin": { "base": { "years": [...], "pop": [...],
                             "class": ["decline", "strong-decline", ...] }, ... }
  }
}
```

Уровни 0–1 (страна, области, Минск, облцентры) несут квантили q10/q90;
районы и города — точечную траекторию до 2050 г. и **классы динамики**
(`growth / stable / decline / strong-decline`) после 2050 г.

**UI-требования (правила честности, обязательны к реализации):**

1. Слайдер продлевается до 2075; зона прогноза визуально отделена
   (штриховка/пунктир фона, метка «прогноз»).
2. Переключатель сценариев (3 кнопки) виден только в прогнозной зоне;
   активный сценарий отображается в легенде и всех тултипах.
3. Графики панелей «Территория» и «Урбанизация» показывают веер q10–q90,
   расширяющийся со временем.
4. Для районов/городов после 2050 г. тултип показывает класс динамики и
   диапазон, **не** точечное число.
5. Каждое прогнозное значение помечено типом `f`, сценарием и версией
   (`прогноз v2026.1, сценарий «базовый»`), в стиле существующих типов
   `c/e/r/m`.
6. Этап 3: переключатель стартового ряда «официальный / скорректированный»
   с поясняющей подсказкой и ссылкой на методику.
7. Страница «Методика прогноза» (см. Р3) доступна с карты в один клик.

**Этапность** (синхронизирована с §6): MVP — страна/области/Минск/облцентры,
3 сценария, веер; этап 2 — районы и города, классы динамики, режим «без
центра» в прогнозе; этап 3 — квантили из полного вероятностного прогона,
переключатель стартового ряда.

### INF-01 `zipf` — Иерархия городов и закон Ципфа ⭐ пилот конвейера

- РВ: как менялось соответствие городской системы Беларуси закону Ципфа в

  Минска (сегодня — 4× Гомеля при «ципфовском» ожидании 2×)?
- Д: ряды 222 городов — **уже в `data.json`**; прогноз городов — после Р1
  этапа 2 (до этого — исторический режим).
- И: pop-stat/Демоскоп (см. SOURCES.md).
- М: rank-size в лог-лог осях; оценка наклона Ципфа (OLS по топ-N, N
  фиксируется и обосновывается) по каждому году-срезу.
- В: анимированный scatter (слайдер лет общий с картой), клик по точке —
  подсветка города на карте; график эволюции наклона.
- О: до 1959 г. состав городов неполон (тип `m`), интерполяция между
  переписями; наклон чувствителен к выбору N — показать чувствительность.
- Ц: научно-популярный сюжет «от сети местечек к макроцефалии»; расширение
  панели урбанизации.
- С: **Н**. Приёмка-специфика: наклон и его дов. интервал воспроизводятся из
  пакета скриптом; чувствительность к N в методблоке.

### INF-02 `aging` — Старение районов

- РВ: где и когда демографическая структура районов делает депопуляцию
  самоподдерживающейся (естественная убыль даже при нулевой миграции)?
- Д: возрастно-половые структуры районов 2009/2019 (WP-F1!), прогнозные
  срезы (Р1); контингенты первоклассников как лидирующий индикатор.
- И: тома переписей; Белстат «Регионы РБ»; госпрограмма
  [«Развитие демографического потенциала»](https://pravo.by/document/?guid=3871&p0=C22500765)
  (реперные доли: 17,1% моложе / 24,5% старше трудоспособного, 01.01.2025).
- М: медианный возраст, доля 65+, коэффициент демографической нагрузки по
  районам; «год пересечения порога 30% доли 65+» из прогноза; пирамиды.
- В: хороплет с переключателем индикатора + пирамида в карточке района
  (2009/2019/прогноз); режим «время до порога».
- О: межпереписные структуры районов — модельные (CCR), у малых районов
  погрешность; свежих (2020+) районных структур нет — сказать явно.
- Ц: высокая; объясняет механизм убыли и наполняет прогноз смыслом; те же
  данные, что у Р1 (синергия).
- С: **С**. Приёмка-специфика: пирамиды сходятся с переписными итогами
  (тест); «год порога» подписан сценарием.

### INF-03 `wages` — Зарплата × динамика населения

- РВ: следует ли население за деньгами: какова эластичность десятилетней
  динамики населения района по зарплатному дифференциалу к Минску? Где
  аномалии (моногорода: зарплата высокая — убыль; агрогородки: наоборот)?
- Д: средняя номинальная зарплата по районам (помесячно, 2010–2026),
  занятые в экономике по районам; ВРП по областям; ряды населения проекта.
- И: [dataportal.belstat.gov.by](https://dataportal.belstat.gov.by/),
  сборник «Регионы Республики Беларусь».
- М: биваритная классификация 3×3 (зарплатный дифференциал × динамика
  населения); панельная регрессия динамики на дифференциал с контролями
  (размер центра, доступность из INF-04) — коэффициент идёт в гравитационную
  модель Р1.
- В: bivariate choropleth + scatter-панель с подсветкой квадрантов и
  выбросов; слой ВРП на душу (области).
- О: зарплата по месту работы, не жительства (маятник вокруг Минска);
  номинал → дефлировать; ВРП только областной.
- Ц: высокая; прямая проверка экономической гипотезы депопуляции; общий
  механизм с прогнозом.
- С: **С**. Приёмка-специфика: регрессия воспроизводится из пакета; в
  методблоке — устойчивость коэффициента к спецификации.

### INF-05 `migration` — Внутренняя и внешняя миграция

## 3. Р3. Методологическая прозрачность

**Правило публикации: инфографика без методологического блока и пакета
артефактов не деплоится.** Проверяется в CI (см. Р4): для каждого slug из
реестра инфографик должны существовать `web/content/methods/<slug>.md` и
собранный пакет.

Методологический блок — единый шаблон `docs/templates/method-block.md`,
обязательные поля (в этом порядке):

1. **Данные и источники** — что, откуда, лицензия, прямые ссылки.
2. **Период и актуальность** — временной охват; дата обращения к каждому
   источнику; дата последнего обновления инфографики.
3. **Преобразования** — гармонизация, интерполяции, дефлирование, зональная
   статистика… (по шагам, со ссылкой на код в пакете).
4. **Модели и допущения** — какие модели, ключевые параметры, почему.
5. **Валидация** — как проверялся результат (тесты, бэктест, сверка с
   внешними оценками), с числами.
6. **Сильные стороны** — что этот подход надёжно показывает.
7. **Ограничения** — известные слабости, где выводы применять нельзя.
8. **Устойчивость** — как меняются выводы при неполных/противоречивых
   данных (чувствительность); что именно неустойчиво.

UI: кнопка «О данных и методике» на каждой инфографике (drawer/страница);
внизу блока — ссылка «Скачать проверяемый пакет (ZIP)» и версия исследования.
Существующие карта и панели тоже получают такой блок (ретроактивно, на базе
METHODOLOGY.md/SOURCES.md) — задача РК+ИГ этапа 0.

## 4. Р4. Проверяемые пакеты артефактов

Нормативный документ — **[ARTIFACT_STANDARD.md](ARTIFACT_STANDARD.md)**
(стандарт v1.0: структура ZIP, `manifest.json`, `AGENT.md`, правила для
больших/лицензионно-ограниченных данных, версионирование и цитирование).

Задачи этого блока:

1. **РК: конвейер сборки** `etl/artifacts/` — сборка пакета из декларативного
   описания (`artifacts/<slug>.yaml`: перечень файлов, ролей, entrypoints,
   ожидаемые результаты с допусками), генерация манифеста и контрольных
   сумм, упаковка `by-maps-<slug>-v<X.Y.Z>.zip` в `web/public/artifacts/`.
2. **РК: валидатор** `python -m etl.artifacts.validate <zip>`: схема
   манифеста, целостность chksum, наличие обязательных файлов, сухой прогон
   `run.sh` в чистом окружении (CI-джоб в контейнере), сверка результатов с
   `expected_results` в допусках.
3. **ИГ: наполнение** пакетов для каждой инфографики и прогноза (данные,
   код, параметры, LIMITATIONS, AGENT-инструкции).
4. **CI-гейт**: пайплайн (сейчас pytest → vitest → tsc → build) дополняется
   шагом `artifacts validate --all`; публикация блокируется при провале.
5. Пилот стандарта — пакет INF-01 `zipf` (минимальный по данным), затем
   пакет прогноза (максимальный по сложности) — по ним дорабатывается
   стандарт до v1.1.

## 5. Р5. Раздел лендинга «Проверяемые артефакты» и новая структура сайта

### 5.1. Структура сайта (Next.js, статический экспорт сохраняется)

```
/                       карта (как сейчас) + прогнозный режим (Р1)
/research               галерея исследований: карточки INF-01…INF-09
/research/<slug>        инфографика + методблок + кнопка пакета
/methodology            общая методика (эволюция METHODOLOGY.md)
/artifacts              раздел «Проверяемые артефакты»
/artifacts/<slug>       страница версий пакета конкретного исследования
```

Главная остаётся картой (текущая ценность продукта); новые разделы — в
шапке. Панель «Урбанизация» получает ссылку на INF-01, карточка района — на
INF-02/INF-07 (перекрёстная навигация «карта ↔ исследования»).

### 5.2. Содержание `/artifacts`

1. **Что это** — пакет = данные + код + параметры + проверки + инструкции
   для LLM-агента; схема содержимого (из стандарта).
2. **Зачем** — принцип проекта: передаём не картинку, а проверяемый пакет,
   который ваша агентская система может исследовать, воспроизвести и
   оспорить.
3. **Как скачать и проверить** — пошагово: вручную (README → run.sh →
   сравнить с expected) и агентом (передать ZIP + AGENT.md).
4. **Что поручить агенту** — готовые промпты: «проверь доступность и
   контрольные суммы источников», «повтори расчёт и сравни с
   заявленными результатами», «найди самые сильные допущения и проверь
   чувствительность к ним», «сформируй отчёт о расхождениях».
5. **Как повторить расчёты** — требования к окружению (Python-версия,
   requirements.lock, один вход `run.sh`), ожидаемое время.
6. **Как искать расхождения** — где лежат допущения (`params/`), список
   известных ограничений (LIMITATIONS.md), таблица допусков.
7. **Как цитировать версию** — semver + git-тег + `CITATION.cff`;
   неизменность опубликованных версий (новые данные → новая версия).

**Сценарий-пример (обязателен на странице)**: пользователь скачивает
`by-maps-zipf-v1.0.0.zip` → передаёт своему агенту с промптом из п. 4 → агент
проверяет chksum источников, запускает `run.sh`, сравнивает наклон Ципфа с
заявленным (допуск в манифесте), пробует альтернативный N городов из
LIMITATIONS → выдаёт отчёт: «совпало / разошлось / спорные допущения».
Показать реальный (заранее прогнанный) пример такого отчёта.

### 5.3. Прототип и приёмка раздела

РК делает кликабельный прототип (может быть статикой в `web/`) до вёрстки;
приёмка: пользователь без подготовки за ≤5 минут понимает, как скачать пакет
и что с ним делать (тест на 2–3 внешних читателях).

## 7. Критерии приёмки инфографики (универсальный чек-лист)

**Данные.** Все источники завендорены/зафиксированы с URL, датой обращения,
лицензией; pytest-инварианты на ключевые суммы/сходимость; каждое число в UI
прослеживается до файла в пакете.

**Методология (Р3).** Все 8 полей шаблона заполнены; вычитка РЕД (не
автором); ограничения сформулированы конкретно (не «данные могут быть
неточны», а какие, насколько, в какую сторону).

**Пакет (Р4).** `validate` проходит: схема манифеста, chksum, полный прогон
`run.sh` в чистом контейнере воспроизводит `expected_results` в допусках;
AGENT.md с задачами проверки и форматом отчёта; LIMITATIONS.md непустой.

**UI/продукт.** Работает в статическом экспорте; бюджет данных страницы
≤1 МБ сжатых (иначе — ленивые подгрузки); мобильная вёрстка; тултипы с
типами достоверности; перекрёстные ссылки карта↔исследование; вёрстка
методблока и кнопка пакета на месте.

**Честность неопределённости.** Оценки — интервалами; прогнозные значения —
со сценарием и версией; на дальнем горизонте/малых территориях — классы, не
точки; формулировки не сильнее данных.

**Специфика прогнозной карты (дополнительно).** Гейты WP-F5 пройдены
(бэктест лучше бенчмарка, национальный итог ±2%, сверка с WPP/WCDE
задокументирована); сценарии откалиброваны по WP-F4; правила честности
WP-F6 (пп. 1–7) реализованы полностью.

**Релиз.** Подписи ИГ + РК + РЕД в PR; git-тег версии; changelog пакета;
страница `/artifacts/<slug>` обновлена.
