# Стандарт проверяемого пакета артефактов (v1.0)

Статус: нормативный документ проекта. Дата: 2026-07-11.
Область действия: каждая публикуемая инфографика и прогноз (см.
[TASK_SPEC.md](TASK_SPEC.md), Р4). Пакет без соответствия стандарту не
проходит CI-гейт публикации.

## 1. Назначение и принципы

Пакет передаёт пользователю не готовый вывод, а **данные + код + параметры +
проверки**, достаточные, чтобы человек или его LLM-агент могли:

1. проверить источники (доступность, соответствие контрольным суммам);
2. воспроизвести расчёт от сырых данных до чисел на лендинге;
3. оспорить допущения (найти их, изменить, увидеть чувствительность);
4. сослаться на конкретную неизменяемую версию.

Принципы: самодостаточность (или воспроизводимый сценарий получения
недостающего), машиночитаемость (манифест — источник истины о содержимом),
детерминированность (фиксированные версии, seed'ы, допуски), честность
(ограничения — обязательная часть пакета, а не примечание).

## 2. Имя и размещение

`by-maps-<slug>-v<MAJOR.MINOR.PATCH>.zip`, например `by-maps-zipf-v1.0.0.zip`.
Публикация: `web/public/artifacts/` + страница `/artifacts/<slug>`.
Опубликованная версия неизменяема; любое изменение данных, кода или
параметров → новая версия + запись в changelog.

## 3. Структура пакета

```
by-maps-<slug>-v1.0/
├── README.md               # описание исследования (см. §4)
├── manifest.json           # машиночитаемый манифест (см. §5) — источник истины
├── AGENT.md                # инструкции для LLM-агента (см. §6)
├── LIMITATIONS.md          # известные ограничения и где выводы неприменимы
├── PROVENANCE.md           # происхождение файлов: цепочка источник→преобразование→результат
├── CITATION.cff            # как цитировать (версия, авторы, URL, git-тег)
├── LICENSE.md              # лицензии пакета и каждого источника
├── sources/
│   ├── registry.csv        # реестр источников: id,url,описание,лицензия,дата_обращения,sha256,как_получить
│   └── raw/                # завендоренные исходники (если лицензия и размер позволяют)
├── data/
│   ├── intermediate/       # промежуточные результаты (curated CSV и т.п.)
│   └── final/              # ровно те данные, что показаны на лендинге
├── code/
│   ├── fetch.py            # получение сырых данных (для невендоренных — обязателен)
│   ├── build.py|*.ipynb    # расчёты от sources/ до data/final/
│   ├── requirements.lock   # зафиксированные зависимости + версия Python
│   └── run.sh              # единственная точка входа: воспроизводит всё
├── params/
│   └── assumptions.yaml    # параметры моделей и допущения, каждое с обоснованием и ссылкой
└── checks/
    ├── tests/              # автотесты (инварианты данных, юнит-тесты расчётов)
    ├── expected_results.json  # контрольные значения с допусками (см. §5)
    └── checksums.sha256    # контрольные суммы всех файлов пакета
```

Минимальный обязательный состав (must): README, manifest.json, AGENT.md,
LIMITATIONS.md, LICENSE.md, sources/registry.csv, data/final/, code/run.sh +
requirements.lock, checks/expected_results.json, checks/checksums.sha256.
Остальное — should; отсутствие обосновывается в README.

## 4. README.md

Обязательные разделы: исследовательский вопрос и главный вывод (1 абзац);
что в пакете и как устроено; как воспроизвести (команды, требования к
окружению, ожидаемое время); как интерпретировать результаты и допуски;
ссылка на живую версию на лендинге и на методологический блок; версия и
changelog.

## 5. manifest.json

Схема (JSON Schema хранится в `etl/artifacts/schema/manifest.schema.json`):

```jsonc
{
  "schema_version": "1.0",
  "id": "zipf",
  "title": "Иерархия городов и закон Ципфа, 1897–2026",
  "version": "1.0.0",
  "released": "2026-09-01",
  "git_tag": "artifact-zipf-v1.0.0",
  "landing_url": "https://by-population-maps.vercel.app/research/zipf",
  "authors": ["..."],
  "files": [
    { "path": "data/final/zipf_slopes.csv", "sha256": "…",
      "role": "final",          // raw|intermediate|final|code|params|docs
      "license": "CC BY 4.0" }
  ],
  "sources": [
    { "id": "ps_cities", "url": "https://pop-stat.mashke.org/belarus-cities.htm",
      "accessed": "2026-07-11", "sha256": "…",
      "vendored": "sources/raw/ps_cities.html",
      "fetch": "code/fetch.py --source ps_cities" }
  ],
  "entrypoints": {
    "reproduce": "code/run.sh",
    "verify": "python -m pytest checks/tests/"
  },
  "environment": { "python": "3.12", "lock": "code/requirements.lock", "seed": 42 },
  "expected_results": [
    { "metric": "zipf_slope_2019", "value": -1.18, "tolerance": 0.02,
      "description": "наклон rank-size по топ-50 городов, перепись 2019" }
  ],
  "limitations_file": "LIMITATIONS.md",
  "agent_instructions": "AGENT.md"
}
```

Правила: каждый файл пакета присутствует в `files[]` с sha256 и ролью;
каждый источник — в `sources[]` с датой обращения; `expected_results` —
непустой и покрывает все ключевые числа, показанные на лендинге.

## 6. AGENT.md — инструкции для LLM-агента

Обязательные разделы:

1. **Контекст**: что за исследование, что считается главным результатом.
2. **Задача 1 — проверка источников**: перечень URL и chksum; что делать
   при недоступности (сверить с завендоренной копией) и несовпадении
   (зафиксировать: источник изменился после даты обращения).
3. **Задача 2 — воспроизведение**: команды окружения и `run.sh`; где лежат
   результаты; сравнить с `expected_results` в допусках.
4. **Задача 3 — стресс допущений**: список 3–7 самых сильных допущений
   (со ссылками на `params/assumptions.yaml` и LIMITATIONS.md) и как их
   варьировать (готовые флаги/параметры запуска); какие выводы должны
   остаться устойчивыми, а какие могут «поплыть».
5. **Формат отчёта**: совпадения / расхождения (с величинами) / недоступные
   источники / спорные допущения / вердикт воспроизводимости.
6. **Границы**: чего пакет не утверждает (из LIMITATIONS.md) — чтобы агент
   не «опровергал» то, что и не заявлялось.

## 7. Большие и лицензионно-ограниченные данные

Если файл нельзя вложить (лицензия, размер > ~50 МБ):

1. в `sources[]` остаются URL, метаданные, лицензия, дата обращения,
   sha256 полученного файла;
2. `code/fetch.py` воспроизводимо скачивает и проверяет chksum;
3. в README — явное предупреждение, что для полного воспроизведения нужен
   доступ к источнику; если источник может исчезнуть — минимальная
   производная выжимка (агрегат), достаточная для проверки расчётов,
   вкладывается в `data/intermediate/` с описанием, как она получена.

Для растров (VIIRS и т.п.): фиксируется версия продукта и тайлы; в пакет
вкладываются зональные агрегаты, скрипт скачивания — для полной проверки.

## 8. Версионирование и цитирование

Semver: MAJOR — изменение выводов/методики, MINOR — обновление данных без
смены выводов, PATCH — исправления, не влияющие на числа. Каждый релиз:
git-тег `artifact-<slug>-v<X.Y.Z>`, строка в `CHANGELOG.md` пакета, обновление
`/artifacts/<slug>`. CITATION.cff указывает точную версию; опционально —
депонирование релизов на Zenodo с DOI (решение на этапе 1 по пилоту).

## 9. Валидация и CI

`python -m etl.artifacts.validate <zip|--all>` проверяет: схему манифеста;
соответствие файлов и chksum; обязательный состав (§3); прогон `run.sh` в
чистом контейнере (фиксированный образ) и сходимость с `expected_results`;
наличие непустых LIMITATIONS.md и AGENT.md. CI: шаг `artifacts validate
--all` после существующих pytest → vitest → tsc → build; провал блокирует
деплой. Локально: `make artifact SLUG=zipf` собирает и валидирует пакет.

Чек-лист приёмки пакета — в [TASK_SPEC.md §7](TASK_SPEC.md). Пилоты
