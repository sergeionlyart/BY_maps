# Предрелизное аудиторское заключение — BY Maps

**Объекты аудита:** репозиторий `BY_maps` (последний коммит — «Этап 8: ML-challenger»), опубликованный сайт https://by-population-maps.vercel.app, исследовательские материалы, данные, расчёты и скачиваемые артефакты.
**Дата:** 12.07.2026. **Роль:** независимый технический и исследовательский рецензент.
**Позиция аудита:** оцениваем только готовность к первому публичному релизу, а не идеальность проекта.

---

## 1. Общая оценка готовности

Проект находится в **высокой степени готовности**. Это зрелая, методологически дисциплинированная работа: воспроизводимый ETL, 155 проходящих Python-тестов, статический сайт без ошибок рендеринга, 10 исследований с обязательными методблоками, прогноз с бэктестами и внешней калибровкой, а также редкий для публицистических проектов стандарт «проверяемых пакетов артефактов» (данные + код + допущения + инструкции для LLM-агента), причём **все 11 пакетов воспроизводятся байт-в-байт** (проверено в чистом окружении).

Внешняя сверка подтвердила фактологию: ключевые цифры совпадают с официальными источниками **вплоть до человека** (см. §2). Существенных ошибок в расчётах и методологии не обнаружено. Найденные блокеры — это ошибки согласованности (устаревшие версии в текстах, красный тест, недоступный репозиторий), устраняемые за 1–2 дня, а не содержательные дефекты.

## 3. Критические замечания (устранить до публикации)

**C-1. Публично недоступен GitHub-репозиторий.** `github.com/sergeionlyart/BY_maps` отдаёт 404 (приватный?), при этом ссылки на него стоят практически на каждой странице сайта и в каждом пакете (METHODOLOGY, SOURCES, validation.md, git-теги релизов). Это подрывает центральный тезис проекта — «ни один вывод не требует доверия на слово». Верифицируемость сейчас декларирована, но внешне невыполнима в части кода и тегов.

**C-2. Красный тестовый гейт в web.** `lib/research.test.ts` устарел после добавления 10-го исследования: ожидает ровно 9 записей и имя файла `by-maps-<slug>-…` (у ML slug `ml`, файл `mlchallenger`). Заявленный CI-гейт (pytest → vitest → tsc → build) на текущем коммите красный — релизить с красным гейтом нельзя, тем более что гейт — часть публичной методологии.

**C-3. Рассинхронизация версий в видимом контенте.** Страница методики прогноза (`content/methods/forecast.md`) объявляет **v2026.3** и ссылается на пакет **v1.2.0**, тогда как опубликованы forecast.json **v2026.4** и пакет **v1.3.0**; шапка `/research/aging` показывает **v1.0.2** при актуальном пакете **v1.0.3**. Читатель, проверяющий проект, скачает не ту версию, которую описывает текст, — для проекта о проверяемости это критично.

**C-4. Инструкция воспроизведения в README невыполнима.** `pip install shapely pytest` недостаточно: тесты падают без `pyyaml` (CI ставит его, README — нет). Формулировка «без внешних зависимостей, кроме shapely» неточна (pyyaml обязателен; numpy/rasterio/pdfplumber нужны для отдельных ETL-шагов nightlights/wpf1).

**C-5. «49 пар» моногородов против 46 в данных.** SOURCES.md и `methods/monotowns.md` заявляют реестр из 49 пар «город—предприятие», фактически в `registry.json`/расчётах — 46. Расхождение «текст против данных» в опубликованном исследовании.

**C-6. Противоречие в площади/плотности Минска.** METHODOLOGY.md и `methods/map.md` описывают полигон Минска ~216 км² и делают вывод «плотность города завышена», но в поставляемом `data.json` площадь Минска — **409,5 км²** (официальная ~349 км²), т.е. плотность фактически **занижена** — оговорка инвертирована относительно данных. Нужно проверить полигон и переписать оговорку (и пересчитать плотность, если полигон ошибочен).
