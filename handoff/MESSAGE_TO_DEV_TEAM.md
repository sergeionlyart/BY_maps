# Группе разработки: доработка BY Maps до релиза 1.0

Коллеги, добрый день.

Завершён независимый предрелизный аудит проекта (репозиторий + опубликованный сайт + данные + артефакты) и подготовлен полный пакет материалов для доработки. Вердикт аудита: **проект публикуем после исправлений** — методологических проблем нет, все блокеры локальные, суммарно ~1–2 дня работы плюс внедрение новых разделов.

Все материалы лежат в папке **`handoff/`** в корне репозитория. Ниже — что делать, в каком порядке и на что опираться.

## 1. Порядок изучения материалов

1. **`handoff/00_README.md`** — начните отсюда: обзор пакета, что уже проверено, известные ограничения.
2. **`handoff/01_audit/AUDIT_REPORT.md`** — аудиторское заключение: что проверялось и результаты (внешняя сверка цифр, тесты, живой сайт).
3. **`handoff/01_audit/FIX_TASKS.md`** — **основной рабочий документ**: задачи T-01…T-17 с приоритетами (P0/P1/P2), указанием файлов и критериями приёмки.
4. **`handoff/02_site/SITE_STRUCTURE.md`** — целевая структура сайта, навигация, схема двуязычия RU/BE, требования к реализации страниц, Definition of Done релиза.
5. **`handoff/03_content/ru/`** и **`handoff/03_content/be/`** — готовый контент шести новых страниц (about, goals, methodology, data-artifacts, author, article) на двух языках.
6. **`handoff/04_artifacts/`** — `ARTIFACTS_CATALOG.md` + `catalog.json` (каталог пакетов с sha256) и `REPRODUCTION.md` (процедура воспроизведения).

## 2. Требования к доработке

**Этап A — блокеры релиза (P0, задачи T-01…T-07 из FIX_TASKS):**

1. Открыть публичный доступ к GitHub-репозиторию и запушить все теги (T-01) — сейчас 404, а на репозиторий ссылается каждая страница.
2. Починить `lib/research.test.ts` — 2 устаревших теста, CI-гейт красный (T-02).
3. Синхронизировать версии в контенте: методика прогноза v2026.3→v2026.4, ссылка на пакет v1.2.0→v1.3.0 (T-03); шапка /research/aging v1.0.2→v1.0.3 (T-04).
4. Исправить README: воспроизведение требует `pyyaml` (T-05).
5. Устранить расхождение «49 пар» моногородов против 46 в данных (T-06).
6. Разрешить противоречие площади/плотности Минска: docs 216 км² против 409,5 в data.json при официальных ~349 (T-07).

**Этап B — новые разделы и двуязычие (по SITE_STRUCTURE):**

7. Создать маршруты `/article`, `/about`, `/goals`, `/author`; расширить `/methodology` и `/artifacts`. Контент брать из `03_content/ru/` **без правок цифр и формулировок** — все числа прошли двойную сверку с data.json/forecast.json и внешними источниками.
8. Белорусские версии — маршруты с префиксом `/be/…`, контент из `03_content/be/`. Внутренние ссылки в BE-файлах уже расставлены под эту схему — сохранить её.
9. Рендер строго статический (markdown читается на этапе сборки, как на страницах исследований), НЕ клиентским fetch — /methodology сейчас единственная «пустая» в HTML страница, заодно исправить (T-08).
10. Метаданные страниц брать из HTML-комментария в первой строке каждого md-файла (title | description). Навигация и футер — по §2 SITE_STRUCTURE.
11. Обязательные технические мелочи: якорь `id="reproduction"` на /artifacts; стили для плашек **[Данные]/[Расчёт]/[Модель]/[Интерпретация]**; переключатель RU|BY; hreflang и `<html lang>`.

**Этап C — артефакты:**

12. Положить `catalog.json` в `web/public/artifacts/`; опубликовать внешние sha256 (T-09). **Важно: после любой пересборки пакетов контрольные суммы перегенерировать** — значения в каталоге зафиксированы на 12.07.2026.
13. По остатку времени — P1: T-10…T-12 (сноска об урбанизации-1897, мелкие фактические правки текстов, размеры пакетов).

## 3. На что опираться при внесении изменений

- **Критерии приёмки в FIX_TASKS.md** — задача закрыта только при их выполнении.
- **Definition of Done релиза** — §6 SITE_STRUCTURE.md: зелёный CI, все маршруты со статическим контентом, публичный GitHub, совпадение версий, RU+BE опубликованы.
- **Существующие паттерны репозитория**: компонент `components/Markdown.tsx`, статический рендер по образцу страниц `/research/<slug>`, реестр `web/lib/research.ts` как единый источник версий.
- **Полный CI-прогон перед деплоем**: pytest (155) → vitest → tsc → next build → `etl.artifacts.build --all --check` → `etl.artifacts.validate --all`. Эталонные результаты и команды — в `04_artifacts/REPRODUCTION.md`.
- **Контент не редактировать содержательно.** Опечатки/вёрстка — ок; изменение цифр, формулировок выводов и оговорок — только через автора. Белорусским текстам нужна вычитка носителем до публикации — заложите это в план.
- После деплоя — smoke-тест по чек-листу §2 AUDIT_REPORT (маршруты, скачивание пакетов, карта: слайдер лет, сценарии, «район без центра») и релизный тег `release-1.0`.

Вопросы по любому пункту — welcome; по контенту и методологии финальное слово за автором проекта.
